# BBQ 買い出し支援アプリ 設計書

## 1. 目的

毎年開催される BBQ の買い出しで、昨年のレシートや記憶を頼りに商品を探し、予算内に収まるよう計算機で合計金額を足している作業をアプリ化する。

このアプリでは、昨年の購入履歴から買うものリストを作り、当日の買い物中に商品価格を入力しながらカゴ投入状況と予算残高を確認できるようにする。買い出し完了後は、予算・購入金額・残金または不足額を収支報告として確認できる。

## 2. 想定ユーザー

- BBQ の買い出し担当者
- 買い出しを手伝うメンバー
- 収支を確認する幹事

## 3. 前提条件

- 個人・小規模利用を想定する。
- 独自ドメインは取得しない。
- 無料枠で運用できる構成にする。
- アプリ URL を知っていて、事前に設定された ID とパスワードを持つ人のみログインできる。
- 一度ログインしたユーザーは、明示的にログアウトするまでログイン状態を保持する。
- セキュリティリスクを最小限にするため、パスワードや管理用の秘密鍵をフロントエンドに埋め込まない。

## 4. 推奨技術構成

### フロントエンド

- React
- Vite
- TypeScript
- Tailwind CSS または通常の CSS

### ホスティング

以下のいずれかを利用する。

- Vercel Free
- Cloudflare Pages Free
- GitHub Pages

独自ドメインは使わず、各サービスが発行する無料 URL を使用する。

### バックエンド / データベース / 認証

- Supabase Free
  - Supabase Auth: ログイン管理
  - Supabase Database: BBQ イベント、購入履歴、買い物リスト、購入実績の保存
  - Row Level Security: ログイン済みユーザーのみデータアクセス可能にする

### Supabase を採用する理由

- 無料枠で小規模アプリに十分。
- 認証とデータベースをまとめて扱える。
- パスワードを自前で保存しなくてよい。
- フロントエンドのみの簡易アプリでも、安全なアクセス制御を組みやすい。

## 5. 機能要件

### 5.1 ログイン

- 事前に管理者がユーザーを作成する。
- ユーザーは ID とパスワードでログインする。
- ログイン後はセッションを保持する。
- ログアウトボタンでログアウトできる。

補足:
Supabase Auth はメールアドレス形式のログインを基本とするため、画面上は `user01` のような ID を入力させ、アプリ内部で `user01@bbq.local` のような形式に変換してログインする。

### 5.2 BBQ イベント管理

- 年ごとの BBQ イベントを作成できる。
- イベントには以下を設定する。
  - イベント名
  - 開催年
  - 予算
  - メモ

### 5.3 昨年の購入履歴からリスト作成

- 過去イベントの購入実績を参照する。
- 昨年購入した商品を今年の買い物リストにコピーできる。
- コピー時は、商品名・カテゴリ・昨年価格・数量を引き継ぐ。
- 今年の価格は未入力の状態にする。

### 5.4 買い物リスト編集

- 商品を自由に追加できる。
- 商品名、カテゴリ、予定数量、メモを編集できる。
- 不要な商品は削除または「今回は買わない」にできる。
- 並び替えはカテゴリ順、追加順、購入済み順に対応する。

### 5.5 予算設定

- イベントごとに予算を設定できる。
- 買い物中は以下を常に表示する。
  - 予算
  - 現在の購入合計
  - 残り予算
  - 予算超過額

### 5.6 買い物中の入力

- リストから商品を選び、実際の価格を入力できる。
- リストにない商品をその場で新規追加し、価格を入力できる。
- 数量、単価、税込金額の入力に対応する。
- 入力後、その商品は「カゴに入れた」状態になる。
- カゴから戻すと、チェックを外せる。

### 5.7 チェック状態

- カゴに入れた商品はチェック済みにする。
- チェック済みの商品は購入合計に含める。
- 未チェックの商品は購入予定リストには残るが、購入合計には含めない。

### 5.8 収支報告

買い出し完了後、以下を表示する。

- イベント名
- 予算
- 購入合計
- 残金または不足額
- 購入済み商品一覧
- 未購入商品一覧
- 追加購入した商品一覧
- カテゴリ別合計

必要に応じて、画面をスクリーンショットして共有できる構成にする。MVP では PDF 出力や CSV 出力は必須にしない。

## 7. データ設計

### 7.1 users

Supabase Auth が管理するため、アプリ側では直接パスワードを保存しない。

必要に応じてプロフィール用テーブルを追加する。

```sql
profiles
- id uuid primary key references auth.users(id)
- display_name text not null
- role text not null default 'member'
- created_at timestamptz not null default now()
```

### 7.2 events

```sql
events
- id uuid primary key
- name text not null
- year integer not null
- budget integer not null default 0
- note text
- status text not null default 'active'
- created_at timestamptz not null default now()
- updated_at timestamptz not null default now()
```

status:

- active
- completed

### 7.3 shopping_items

```sql
shopping_items
- id uuid primary key
- event_id uuid not null references events(id)
- name text not null
- category text
- planned_quantity numeric
- actual_quantity numeric
- unit_price integer
- actual_price integer
- last_year_price integer
- note text
- is_checked boolean not null default false
- is_extra boolean not null default false
- sort_order integer not null default 0
- created_at timestamptz not null default now()
- updated_at timestamptz not null default now()
```

項目の意味:

- planned_quantity: 予定数量
- actual_quantity: 実際に買った数量
- unit_price: 単価
- actual_price: 実際に購入合計へ入れる税込金額
- last_year_price: 昨年購入時の価格
- is_checked: カゴに入れたか
- is_extra: 当日追加した商品か

### 7.4 event_members

ログイン済みユーザーのうち、どのイベントを見られるか制御する。

```sql
event_members
- event_id uuid not null references events(id)
- user_id uuid not null references auth.users(id)
- role text not null default 'member'
- created_at timestamptz not null default now()
- primary key (event_id, user_id)
```

role:

- owner
- member

## 8. 権限設計

### 基本方針

- 未ログインユーザーは全データにアクセス不可。
- ログイン済みユーザーは、自分が `event_members` に登録されているイベントのみ参照・編集可能。
- パスワードは Supabase Auth に任せ、アプリ DB には保存しない。
- Supabase の service role key は絶対にフロントエンドに置かない。

### RLS 方針

events:

- select: event_members に自分の user_id がある場合のみ許可
- insert: ログイン済みユーザーのみ許可
- update: event_members に自分の user_id がある場合のみ許可
- delete: owner のみ許可、または MVP では削除機能を実装しない

shopping_items:

- select/insert/update/delete: 紐づく event_id の event_members に自分の user_id がある場合のみ許可

event_members:

- select: 自分が参加しているイベントのみ許可
- insert/update/delete: MVP では Supabase 管理画面から行い、アプリ画面では実装しない

## 9. セキュリティ設計

### 実施する対策

- HTTPS の無料ホスティングを使う。
- パスワードをコードや DB に平文保存しない。
- Supabase Auth でパスワードを管理する。
- Supabase RLS を必ず有効化する。
- フロントエンドに置くのは Supabase anon key のみとする。
- service role key はローカル環境や管理作業でのみ使用し、公開しない。
- ログイン失敗時に「ID が存在しない」「パスワードが違う」などの詳細を出さない。
- 管理者用のユーザー追加は Supabase 管理画面で行い、アプリには管理者画面を作らない。
- 使わなくなったユーザーは Supabase 管理画面で無効化または削除する。

### 残るリスク

- ID とパスワードを知っている人はログインできる。
- 共有端末でログアウトしない場合、次の利用者がアプリを見られる可能性がある。
- 無料サービスの仕様変更や停止に影響を受ける。

### リスク低減策

- 個人ごとに ID を分ける。
- 共通 ID・共通パスワードは避ける。
- BBQ 終了後にパスワードを変更する。
- 年ごとにイベントを分け、不要な古いユーザーは外す。

## 10. 主要ユースケース

### 10.1 事前準備

1. 管理者が Supabase でユーザーを作成する。
2. 今年の BBQ イベントを作成する。
3. 昨年の購入履歴から今年の買い物リストを作成する。
4. 予算を設定する。
5. 必要に応じて商品を追加・削除する。

### 10.2 買い物中

1. 買い出し担当者がログインする。
2. 今年のイベントを開く。
3. 商品を手に取る。
4. リストから商品を選ぶ。
5. 価格を入力する。
6. 商品にチェックが入り、購入合計に反映される。
7. リストにない商品は新規追加して価格を入力する。
8. 予算残高を見ながら買い物を続ける。

### 10.3 買い物完了

1. 収支報告画面を開く。
2. 予算、購入合計、残金または不足額を確認する。
3. 購入済み商品と未購入商品を確認する。
4. 必要に応じて画面を共有する。
5. イベントを完了状態にする。

## 14. 受け入れ条件

- 事前に作成した ID とパスワードでログインできる。
- 未ログイン状態では買い物リストを表示できない。
- 昨年の購入履歴から今年の買い物リストを作成できる。
- リストに商品を追加できる。
- 商品価格を入力すると購入合計に反映される。
- チェック済みの商品だけが購入合計に含まれる。
- 予算との差額が表示される。
- 買い物完了後に収支報告が表示される。
- ログアウトするまでログイン状態が維持される。
- Supabase の RLS が有効で、権限のないイベントデータを取得できない。

## 15. 運用ルール

- ユーザー追加・削除は Supabase 管理画面で行う。
- パスワードは BBQ ごと、または年 1 回変更する。
- 参加者が変わった場合は、不要なユーザーを削除する。
- 予算や購入履歴は年ごとにイベントを分けて管理する。
- レシート原本は必要に応じて別途保管する。
