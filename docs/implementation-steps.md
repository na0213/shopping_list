# BBQ 買い出し支援アプリ 実装手順

この手順書は、作業者が手動で行うことと、Codex に渡すプロンプトを実行順に並べたもの。

実装時は必ず先に [security-rules.md](./security-rules.md) を読み、セキュリティルールを破らないこと。

## 1. 私がやること: Supabase プロジェクトを作成する

Supabase にログインし、新しいプロジェクトを作成する。

設定するもの:

- Project name: `bbq-shopping` など任意
- Database password: 強いパスワードを設定し、ローカルの安全な場所に保管
- Region: 近いリージョンを選択
- Pricing plan: Free

作成後、以下を控える。

- Project URL
- anon public key

注意:

- `service_role` key は控えてもよいが、Codex に渡さない。
- Git に保存しない。
- チャットにも貼らない。

## 2. Codex に渡すプロンプト 1: プロジェクトの土台を作る

```text
docs/security-rules.md と docs/design.md を最初に読んでください。

BBQ 買い出し支援アプリを実装するため、React + Vite + TypeScript のプロジェクト土台を作ってください。

要件:
- セキュリティルールを必ず守ること
- Vite + React + TypeScript
- Supabase クライアントを使う前提の構成
- `.env.example` を作成する
- `.env.local` は作成しない
- `.gitignore` に `.env`, `.env.local`, `.env.*.local`, `node_modules`, `dist` を含める
- まだ実データや実ユーザー情報は入れない
- まずはビルドが通る最小構成にする

完了後、実行したコマンド、作成・変更したファイル、次に必要な手動作業を報告してください。
```

## 3. 私がやること: `.env.local` を作成する

Codex が作成した `.env.example` を参考に、ローカルだけで `.env.local` を作成する。

```env
VITE_SUPABASE_URL=Supabase の Project URL
VITE_SUPABASE_ANON_KEY=Supabase の anon public key
```

注意:

- `.env.local` は Git に入れない。
- `service_role` key は絶対に入れない。
- Codex に実際の値を貼らない。

## 4. Codex に渡すプロンプト 2: DB マイグレーションを作る

```text
docs/security-rules.md と docs/design.md を読んでください。

Supabase 用の DB マイグレーション SQL を作ってください。

要件:
- profiles
- events
- shopping_items
- event_members
- updated_at 自動更新用の trigger
- events / shopping_items / event_members / profiles の RLS 有効化
- event_members によるイベント単位のアクセス制御
- 未ログインユーザーは全データアクセス不可
- 所属していない event_id のデータは select / insert / update / delete 不可
- service_role key を使う前提の実装にしない
- SQL ファイルとして `supabase/migrations/` 配下に作成する

完了後、作成した SQL の内容を要約し、Supabase SQL Editor で実行する順序を説明してください。
```

## 5. 私がやること: Supabase で SQL を実行する

Supabase 管理画面を開き、SQL Editor で Codex が作成したマイグレーション SQL を実行する。

確認すること:

- テーブルが作成されている
- RLS が有効になっている
- Policy が作成されている
- SQL 実行エラーがない

エラーが出た場合:

- エラー全文から秘密情報を除いて Codex に渡す。
- Supabase URL や key は貼らない。

## 6. 私がやること: Supabase Auth でユーザーを作成する

Supabase 管理画面の Authentication から、買い出しメンバー用ユーザーを作成する。

例:

- 画面で使う ID: `user01`
- Supabase Auth の email: `user01@bbq.local`
- password: 自分で決めた強いパスワード

注意:

- ID とパスワード一覧を Git に保存しない。
- Codex にパスワードを渡さない。
- 共通アカウントより、できれば人ごとにアカウントを分ける。

## 7. 私がやること: 初期イベントとメンバーを Supabase に登録する

Supabase SQL Editor で、今年のイベントと `event_members` を登録する。

この作業では、まず Supabase 管理画面で対象ユーザーの `auth.users.id` を確認する。

登録するもの:

- `events` に今年の BBQ イベント
- `event_members` に参加ユーザー
- 必要なら `profiles` に表示名

注意:

- 実ユーザーの UUID や表示名をリポジトリに保存しない。
- SQL をメモとして残す場合は、ローカルの Git 管理外に置く。

## 8. Codex に渡すプロンプト 3: 認証機能を実装する

```text
docs/security-rules.md と docs/design.md を読んでください。

Supabase Auth を使ったログイン・ログアウト機能を実装してください。

要件:
- 画面では ID とパスワードを入力する
- ID は内部で `${id}@bbq.local` 形式に変換してログインする
- パスワードを保存しない
- localStorage にパスワードを入れない
- 未ログイン状態ではログイン画面以外を表示しない
- ログイン失敗時は詳細理由を出さず「ログインできませんでした」と表示する
- ログイン状態は Supabase Auth のセッションに任せる
- ログアウトボタンを実装する
- エラーに secret や token を表示しない

完了後、手元でビルドと動作確認を行い、変更ファイルと確認結果を報告してください。
```

## 9. 私がやること: ログイン動作を確認する

ローカルでアプリを起動し、作成済みユーザーでログインする。

確認すること:

- 正しい ID / パスワードでログインできる
- 間違った ID / パスワードではログインできない
- エラー表示が詳細すぎない
- ログアウトできる
- 再読み込みしてもログイン状態が維持される

問題があれば、表示されたエラーから秘密情報を除いて Codex に渡す。

## 10. Codex に渡すプロンプト 4: イベント一覧と作成を実装する

```text
docs/security-rules.md と docs/design.md を読んでください。

ログイン後のイベント一覧画面とイベント作成機能を実装してください。

要件:
- ログイン済みユーザーが所属するイベントだけを表示する
- RLS 前提で Supabase から取得する
- フロントエンド側で全イベント取得後に絞り込む実装は禁止

- イベント作成時は、作成者を event_members の owner として登録する処理も考慮する
- 予算は 0 以上の整数
- イベント名は必須
- エラー表示に内部情報を出さない

今回は共有ID 1つで運用する前提です。
ただし実装は複数ユーザー・複数イベントでも RLS により安全に動く構成にしてください。
テストデータ、実ID、パスワードはコードに入れないでください。

完了後、ビルドと動作確認を行い、変更ファイルと確認結果を報告してください。
```

## 11. 私がやること: イベント表示と RLS を確認する

Supabase で複数ユーザーを作成し、所属イベントを分けて確認する。

確認すること:

- ユーザー A は自分のイベントだけ見える
- ユーザー B は自分のイベントだけ見える
- 所属していないイベントが表示されない
- 未ログインではイベントが表示されない

可能であれば、Supabase SQL Editor やアプリ画面で RLS の効き方も確認する。


## 追加
docs/security-rules.md と docs/design.md を読んでください。

買い物リスト画面のUIと動作を、当日の買い物中に素早く使える形へ変更してください。

変更したい方針:
- 過去イベントから商品をコピーする機能は使わないため、画面上の導線から外してください
- 商品は事前登録、またはスーパーでその場で追加できるようにしてください
- 商品名の隣にチェックボックスを表示してください
- 商品ごとに数量、税込価格、税抜価格を入力できるようにしてください

- 税抜価格を入力したら税込価格を自動計算してください
- 金額を入力した時点で、自動的に is_checked を true にしてください
- チェック済みの商品だけを購入合計に含めてください
- チェックを外した場合は、購入合計から除外してください
- 複数人で同じイベントを開いたとき、誰かが金額入力またはチェック変更した内容が他の画面にも反映されるようにしてください
  - Supabase Realtime を使える場合は使ってください
  - 難しい場合は更新ボタンで再取得できる状態を維持してください
- 所属していない event_id の商品は表示・追加・更新・削除できないこと
- フロントエンド側で全商品取得後に絞り込む実装は禁止
- React の通常表示を使い、dangerouslySetInnerHTML は使わない

- service_role key を使う前提にしない
- 画面上部または買い物リスト付近に、予算、購入済み合計、あと購入できる金額を常に表示してください
- あと購入できる金額は「イベント予算 - チェック済み商品の税込価格合計」で計算してください
- 金額入力により is_checked が true になったら、購入済み合計とあと購入できる金額へ即時反映してください
- 予算を超えた場合は、あと購入できる金額を 0 円にせず、不足額が分かる表示にしてください
  - 例: あと購入できる金額 -1,200円
  - または 1,200円超過
- 未チェックの商品は購入済み合計に含めないでください

完了後、変更したファイル、DB migration が必要だったか、ビルド結果、動作確認結果を報告してください。
