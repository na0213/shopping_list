# セキュリティルール

このファイルは、BBQ 買い出し支援アプリを Codex で実装するときに最初に読むこと。

このアプリは簡易的な小規模アプリだが、ログイン情報、購入履歴、予算、参加者情報を扱うため、実装では「無料で作る」ことよりも「公開してはいけない情報を公開しない」ことを優先する。

## 1. 最重要方針

- パスワードをアプリコード、設定ファイル、DB テーブル、README、サンプルデータに平文で保存しない。
- Supabase の `service_role` key をフロントエンドに置かない。
- `.env`、`.env.local`、秘密鍵、実ユーザーの ID・パスワードを Git 管理しない。
- 未ログインユーザーに BBQ データを表示しない。
- ログイン済みユーザーでも、参加していないイベントのデータを表示・更新・削除させない。
- Supabase の Row Level Security を必ず有効にする。
- セキュリティを下げる実装で一時的に動かすことを禁止する。

## 2. 禁止事項

以下は実装してはいけない。

- フロントエンド内に固定パスワードを書く。
- `localStorage` にパスワードを保存する。
- ユーザー認証を自作する。
- DB にパスワード用カラムを作る。
- Supabase の `service_role` key を `.env` 以外に書く。
- Supabase の `service_role` key をブラウザで使う。
- RLS を無効にしたまま実装完了とする。
- 全イベントを無条件で取得するクエリを書く。
- ログインしていない状態でデータ取得を実行する。
- エラー内容に内部情報、SQL、キー、認証詳細を表示する。
- ログイン失敗時に「ID が存在しない」「パスワードが違う」を出し分ける。
- 実在する ID、パスワード、予算、購入履歴をサンプルとしてコミットする。
- CORS や認証を緩めるための暫定コードを残す。
- 本番データを使ったテストコードをコミットする。
- 依存パッケージの脆弱性警告を無視して完了扱いにする。

## 3. 認証ルール

- 認証は Supabase Auth を使用する。
- アプリ独自のパスワード照合は実装しない。
- 画面上の ID は、内部的にメールアドレス形式へ変換して Supabase Auth に渡す。
- 例: `user01` を `user01@bbq.local` として扱う。
- ログイン状態は Supabase Auth のセッション管理に任せる。
- ログアウト時は Supabase Auth の signOut を必ず呼ぶ。
- ログイン前の画面では、イベント名、予算、商品名などの業務データを表示しない。

## 4. 環境変数ルール

フロントエンドで使ってよい値:

- Supabase Project URL
- Supabase anon public key

フロントエンドで使ってはいけない値:

- Supabase service role key
- DB 接続文字列
- 管理者パスワード
- 実ユーザーパスワード
- API の秘密鍵

`.env.example` にはダミー値のみを書く。

```env
VITE_SUPABASE_URL=https://example.supabase.co
VITE_SUPABASE_ANON_KEY=example-anon-key
```

`.env.local` は Git 管理しない。

## 5. データアクセスルール

- すべてのイベントデータは `event_members` を通してアクセス制御する。
- アプリ側の絞り込みだけに依存しない。
- DB 側の RLS で必ず制限する。
- `events`、`shopping_items`、`event_members` は RLS を有効化する。
- ユーザーが所属していない `event_id` のデータは select / insert / update / delete すべて不可にする。
- 削除機能は MVP では最小限にし、必要なら論理削除を優先する。

## 6. RLS 必須方針

RLS は以下の考え方で実装する。

### events

- select: 自分が `event_members` に登録されているイベントのみ許可。
- insert: ログイン済みユーザーのみ許可。作成者を `event_members.owner` に登録する処理を必ず用意する。
- update: 自分が `event_members` に登録されているイベントのみ許可。
- delete: 原則実装しない。実装する場合は owner のみ許可。

### shopping_items

- select: 紐づくイベントのメンバーのみ許可。
- insert: 紐づくイベントのメンバーのみ許可。
- update: 紐づくイベントのメンバーのみ許可。
- delete: 紐づくイベントのメンバーのみ許可。ただし MVP では削除より非表示・対象外フラグを優先する。

### event_members

- select: 自分が所属するイベントのメンバー情報のみ許可。
- insert/update/delete: MVP ではアプリから実行しない。管理作業は Supabase 管理画面で行う。

## 7. 入力値ルール

- 商品名、カテゴリ、メモは空文字や異常に長い文字列を制限する。
- 金額、予算、数量は数値として検証する。
- 金額と予算は 0 以上の整数にする。
- 数量は 0 以上の数値にする。
- 表示時は React の通常のテキスト表示を使い、HTML を直接挿入しない。
- `dangerouslySetInnerHTML` は使わない。
- ユーザー入力値を SQL 文字列へ直接連結しない。

## 8. エラー表示ルール

ユーザーに表示してよい内容:

- ログインできませんでした。
- データの取得に失敗しました。
- 保存に失敗しました。
- 入力内容を確認してください。

ユーザーに表示してはいけない内容:

- SQL 文
- Supabase の詳細レスポンス全文
- stack trace
- access token
- refresh token
- API key
- DB 内部構造の詳細

開発中に console へ出す場合も、秘密情報が含まれないようにする。本番向けコードでは過剰な `console.log` を残さない。

## 9. セッション管理ルール

- ログイン状態の維持は Supabase Auth に任せる。
- パスワードや独自トークンを保存しない。
- ログアウトボタンを常に到達可能な場所に置く。
- 共有端末を想定し、ログアウトの導線を分かりやすくする。
- セッション切れ時は自動的にログイン画面へ戻す。

## 10. 依存関係ルール

- 不要なライブラリを追加しない。
- 認証、暗号化、DB アクセスのために出所不明なライブラリを使わない。
- 依存関係を追加した場合は、追加理由を明確にする。
- `npm audit` または同等の確認で重大な脆弱性がないことを確認する。
- 脆弱性が残る場合は、内容と影響を明記して完了報告する。

## 11. Git 管理ルール

コミットしてよいもの:

- ソースコード
- マイグレーション SQL
- `.env.example`
- ドキュメント
- ダミーデータ

コミットしてはいけないもの:

- `.env`
- `.env.local`
- 実ユーザーの ID・パスワード一覧
- Supabase service role key
- DB 接続文字列
- 実レシート画像
- 実際の購入履歴を含む個人データ

`.gitignore` には最低限以下を含める。

```gitignore
.env
.env.local
.env.*.local
node_modules
dist
```

## 12. 無料ホスティング運用ルール

- 独自ドメインは使わず、ホスティングサービスの無料 URL を使う。
- デプロイ先の環境変数には anon key のみ設定する。
- service role key はデプロイ先に設定しない。
- プレビュー URL が第三者に見られても、ログインなしでデータが見えない状態にする。
- URL を知っているだけではデータへアクセスできないことを確認する。

## 13. 実装時チェックリスト

実装完了前に必ず確認する。

- [ ] `.env.local` が Git 管理されていない。
- [ ] `service_role` key がリポジトリ内に存在しない。
- [ ] パスワードらしき文字列がコードに直書きされていない。
- [ ] 未ログイン状態でイベント一覧を取得していない。
- [ ] 未ログイン状態ではログイン画面以外を表示しない。
- [ ] RLS が有効になっている。
- [ ] 所属していないイベントを取得できない。
- [ ] 商品追加・編集時に入力値検証がある。
- [ ] 金額・予算が負数にならない。
- [ ] `dangerouslySetInnerHTML` を使っていない。
- [ ] ログイン失敗理由を詳細に表示していない。
- [ ] エラー表示に内部情報を出していない。
- [ ] ログアウトできる。
- [ ] 依存関係の脆弱性を確認した。

## 14. レビュー時の確認観点

コードレビューでは以下を優先して見る。

- 認証前にデータを読んでいないか。
- DB 側の RLS に頼れる設計になっているか。
- フロントエンドだけの権限制御になっていないか。
- 秘密情報がリポジトリに混入していないか。
- ユーザー入力を安全に扱っているか。
- 失敗時に情報を出しすぎていないか。
- 無料ホスティングの公開 URL で、未ログイン閲覧できないか。

## 15. 判断に迷った場合

判断に迷った場合は、便利さよりも安全側に倒す。

特に以下に該当する場合は、実装を進める前に設計を見直す。

- 秘密情報をフロントエンドに置く必要がありそうな場合。
- RLS を無効にしないと動かない場合。
- 管理者権限をブラウザから使う必要がありそうな場合。
- 実データをサンプルとして使いたくなった場合。
- エラー解決のために認証や権限を緩めたくなった場合。

このアプリでは、多少手間が増えても、認証・権限・秘密情報の扱いを崩さないことを優先する。
